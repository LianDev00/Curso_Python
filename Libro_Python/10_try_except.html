Los errores son inevitables. Esta guía cubre cómo anticiparlos, atraparlos, clasificarlos, relanzarlos y crear los tuyos propios.

try: código que puede fallar OBLIGATORIO
except: qué hacer si falla OBLIGATORIO (al menos uno)
else: qué hacer si NO falló nada opcional
finally: se ejecuta SIEMPRE, pase lo que pase opcional
try
¿Por qué manejar errores?
excepciones · flujo de control · defensivo vs agresivo

Cuando Python encuentra algo que no puede ejecutar — dividir por cero, convertir texto a número, abrir un archivo que no existe — lanza una excepción. Sin manejo, el programa se detiene. Con try/except, interceptas ese error y decides qué hacer.

Sin manejo — el programa muere
n = int(input("Número: "))
# usuario escribe "hola"
# ↓
ValueError: invalid literal
for int() with base 10: 'hola'
# programa terminado
Con manejo — el programa continúa
try:
    n = int(input("Número: "))
except ValueError:
    print("Eso no es un número")
# programa sigue corriendo
Filosofía Python — EAFP vs LBYL:
LBYL (Look Before You Leap) — verificar antes de hacer: if key in dict: use dict[key]
EAFP (Easier to Ask Forgiveness than Permission) — intentar y atrapar el error.
Python favorece EAFP. try/except es la herramienta central de ese estilo.

Estructura completa con los 4 bloques
try:
    # código que puede lanzar excepción
    resultado = 10 / int(input("Divisor: "))

except ZeroDivisionError:
    # solo si ocurre ZeroDivisionError
    print("No se puede dividir por cero")

except ValueError:
    # solo si ocurre ValueError
    print("Debes ingresar un número")

else:
    # solo si el try terminó SIN errores
    print(f"Resultado: {resultado:.2f}")

finally:
    # se ejecuta SIEMPRE — con o sin error
    print("Operación finalizada")
except con alias "as e" — capturar el mensaje del error
try:
    x = int("texto")

except ValueError as e:
    print(f"Error capturado: {e}")
    # → Error capturado: invalid literal for int() with base 10: 'texto'
    print(type(e))
    # → <class 'ValueError'>
    print(e.args)
    # → ("invalid literal for int() with base 10: 'texto'",)
else vs poner código al final del try:
Código en else solo corre si el try fue exitoso. Si lo pones al final del try, también se ejecutaría si ocurre un error y hay un except que lo atrapa. else es más preciso.
finally siempre corre — incluso si hay un return, un break, o si el except relanza el error. Su uso principal: cerrar recursos (archivos, conexiones de base de datos) sin importar si algo falló.

La jerarquía de excepciones en Python — todas heredan de BaseException:

BaseException └── Exception ← casi siempre capturas desde aquí abajo ├── ValueError ├── TypeError ├── AttributeError ├── NameError ├── RuntimeError │ └── RecursionError ├── ArithmeticError │ ├── ZeroDivisionError │ └── OverflowError ├── LookupError │ ├── IndexError │ └── KeyError └── OSError ├── FileNotFoundError └── PermissionError
Nunca atrapes Exception o BaseException sin motivo. except Exception captura absolutamente todo — incluyendo errores de programación que deberías ver y corregir. Siempre especifica el tipo exacto que esperas.

Múltiples except
atrapar diferentes errores · orden importa

Puedes encadenar varios except para manejar distintos tipos de error de forma diferente. Python los evalúa en orden y entra al primero que coincida.

Múltiples except independientes
try:
    indice    = int(input("Índice: "))
    lista     = [10, 20, 30]
    resultado = 100 / lista[indice]

except ValueError:
    print("El índice debe ser un número entero")

except IndexError:
    print("Índice fuera de rango (0–2)")

except ZeroDivisionError:
    print("El valor en esa posición es cero")

else:
    print(f"100 / lista[{indice}] = {resultado:.2f}")
Atrapar múltiples errores en un solo except con tupla
try:
    n = int(input("Número: "))
    r = 100 / n

except (ValueError, ZeroDivisionError) as e:
    # ambos errores van al mismo bloque
    print(f"Entrada inválida: {e}")
El orden de los except importa. Siempre pon los más específicos primero. Si pones except Exception antes que except ValueError, el ValueError nunca será alcanzado — Exception lo atrapará primero porque ValueError hereda de Exception.
Orden incorrecto vs correcto
# ✗ INCORRECTO — ValueError nunca se alcanza
try:
    int("abc")
except Exception:          # atrapa TODO
    print("error general")
except ValueError:         # nunca llega aquí
    print("error de valor")

# ✓ CORRECTO — específico primero
try:
    int("abc")
except ValueError:         # más específico
    print("error de valor")
except Exception:          # captura el resto
    print("otro error")

raise
lanzar excepciones · relanzar · raise from

raise lanza una excepción manualmente. Se usa para señalar condiciones inválidas que Python no detectaría por sí solo, o para relanzar un error atrapado después de hacer algo con él.

raise — lanzar una excepción nueva
def dividir(a, b):
    if b == 0:
        raise ValueError("El divisor no puede ser cero")
    return a / b

try:
    dividir(10, 0)
except ValueError as e:
    print(e)      # → El divisor no puede ser cero
raise sin argumentos — relanzar el error actual
try:
    x = int("texto")

except ValueError as e:
    print(f"Log: ocurrió ValueError → {e}")
    raise           # relanza el mismo error hacia arriba
                    # útil para loggear y que el error siga propagándose
raise ... from — encadenar excepciones con contexto
def cargar_config(archivo):
    try:
        f = open(archivo)
    except FileNotFoundError as e:
        raise RuntimeError(f"No se pudo iniciar: falta {archivo}") from e
        # el traceback mostrará AMBOS errores — el original y el nuevo

# traceback resultante:
# FileNotFoundError: [Errno 2] No such file or directory: 'config.txt'
# The above exception was the direct cause of the following exception:
# RuntimeError: No se pudo iniciar: falta config.txt
raise from None — si quieres lanzar un error nuevo sin mostrar el original en el traceback: raise RuntimeError("mensaje") from None. Limpia el output pero oculta el contexto original.

Excepciones propias
crear clases de excepción personalizadas

Puedes crear tus propias excepciones heredando de Exception. Son útiles para dar nombres semánticos a errores específicos de tu dominio — en lugar de usar un ValueError genérico, usas EdadInvalidaError y queda claro qué falló.

Excepción personalizada básica
# solo heredar de Exception — con eso alcanza
class EdadInvalidaError(Exception):
    pass

def registrar(edad):
    if edad < 0 or edad > 120:
        raise EdadInvalidaError(f"Edad fuera de rango: {edad}")
    print(f"Registrado con edad {edad}")

try:
    registrar(-5)
except EdadInvalidaError as e:
    print(f"Error: {e}")
    # → Error: Edad fuera de rango: -5
Excepción con atributos adicionales
class SaldoInsuficienteError(Exception):
    def __init__(self, saldo, monto):
        self.saldo = saldo
        self.monto = monto
        super().__init__(
            f"Saldo insuficiente: tienes {saldo}, intentas retirar {monto}"
        )

def retirar(saldo, monto):
    if monto > saldo:
        raise SaldoInsuficienteError(saldo, monto)
    return saldo - monto

try:
    retirar(100, 250)
except SaldoInsuficienteError as e:
    print(e)                       # mensaje completo
    print(f"Faltaron: {e.monto - e.saldo}")  # acceso a atributos

Patrones y casos especiales
input seguro · loop con retry · context manager · assert
Patrón: pedir input hasta obtener un valor válido
def pedir_entero(mensaje):
    while True:
        try:
            return int(input(mensaje))
        except ValueError:
            print("Ingresa solo números enteros.")

edad = pedir_entero("Tu edad: ")
# no sale del while hasta que el usuario ingrese un int válido
Patrón: with statement — context manager (cierre automático)
# sin with — el archivo puede quedar abierto si hay error
f = open("datos.txt")
try:
    contenido = f.read()
finally:
    f.close()    # cierre manual en finally

# con with — cierre automático, más limpio
with open("datos.txt") as f:
    contenido = f.read()
# el archivo se cierra solo al salir del bloque, incluso si hay excepción

# combinado con try/except
try:
    with open("datos.txt") as f:
        contenido = f.read()
except FileNotFoundError:
    print("Archivo no encontrado")
assert — verificaciones de desarrollo
# assert condicion, "mensaje si falla"
# lanza AssertionError si la condición es False

def calcular_promedio(notas):
    assert len(notas) > 0, "La lista no puede estar vacía"
    return sum(notas) / len(notas)

calcular_promedio([])
# → AssertionError: La lista no puede estar vacía

# assert es para errores de PROGRAMADOR, no de usuario
# no uses assert para validar input del usuario
# los assert se pueden desactivar con python -O (modo optimizado)
Caso especial: except en bucle — continuar vs romper
datos = ["10", "abc", "30", "0", "50"]
resultados = []

for dato in datos:
    try:
        n = int(dato)
        r = 100 / n
        resultados.append(r)
    except ValueError:
        print(f"'{dato}' no es número — saltando")
        continue   # salta esta iteración, sigue con la siguiente
    except ZeroDivisionError:
        print("División por cero — deteniendo")
        break      # sale del for completamente

print(resultados)    # → [10.0] — solo '10' fue válido antes del 0
Caso especial: except con finally y return
def demo():
    try:
        print("try")
        return "desde try"     # parece que retorna aquí...
    finally:
        print("finally")       # ...pero finally corre ANTES del return
        # si pones return aquí, SOBREESCRIBE el return del try

resultado = demo()
# imprime: "try"
# imprime: "finally"
# resultado = "desde try"

# TRAMPA: return dentro de finally anula el return del try
def trampa():
    try:
        return "try"
    finally:
        return "finally"   # este gana siempre

trampa()  # → "finally"  — evita esto
Casos que confunden:
· except: sin tipo — atrapa TODO incluyendo KeyboardInterrupt y SystemExit. Casi nunca lo quieres.
· try/except vacío con pass — silencia errores sin hacer nada. Muy difícil de debuggear después.
· return dentro de finally — sobreescribe el return del try. Evítalo.
· assert con tupla — assert (False, "msg") siempre pasa porque una tupla es truthy. Usa assert False, "msg".
